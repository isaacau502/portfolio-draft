# PART I — DESIGN DOC

## Purpose

A fourth search operator, peer to `improve` / `combine` / `repair`, whose job is to make
the LLM produce a **completely novel solution** — a brand-new architecture or approach for
the same objective — rather than refine the current candidate. It exists to break the
**template lock-in** observed across runs: the LLM stays inside the `DecisionTreeClassifier`
family because the seed and prior candidates are heavy tree-specific code, so it refactors
rather than reimagines. The novelty operator is the *explore* counterpart to the three
existing *exploit* operators.

**Distinct from multi-init:** multi-init diversifies the *starting* population (fresh
seeds). The novelty operator injects diversity *mid-search*, carrying the accumulated
context of the run so far. Firing it at try 80 leverages 80 tries of learning instead of
starting from scratch — that mid-search context inheritance is the whole reason it earns a
place alongside multi-init rather than duplicating it.

## Locked decisions

### L1 — Operator intent
Instruct the LLM to produce a *completely novel solution* — a brand-new architecture being
the canonical example, but framing intentionally open-ended (new architecture, new
approach, or something unanticipated; the LLM decides what "novel" means).
- **Domain-free:** never names specific model families (no "try RandomForest/SVM"). The LLM
  must discover *which* architecture itself.
- **Operation-directive:** explicitly demands a departure from the current solution. (Note:
  this is directive, not "content-free" in the earlier sense — it tells the LLM to leave
  the family, without naming the destination.)
- **High variance accepted:** a maximally open novelty prompt trades reliability for
  exploration. Brilliance-or-garbage is expected and acceptable, because the operator's job
  is exploration, not dependable improvement.

### L2 — Success criterion (this project)
Judged by **eye test** via the run-summary feature. The specific bar is sharper than
"explored diversity": **did the operator's output replace the core with a genuinely
different architecture, versus a cosmetic refactor of the parent's existing model?**
Same-family output = the mechanism failed, regardless of how novel the rationale prose reads.
- This is an **exploration/diversity check, explicitly NOT a frontier-improvement claim.**
  The operator "succeeding" means it produced non-DecisionTree solutions — not that they
  landed on the frontier.
- Quantitative grading (hypervolume-per-dollar) is a **separate, later project** (see
  Out-of-scope). This operator ships and is judged qualitatively first.
- **Operationalized via analysis-prompt change (required, out of code scope):** the run
  summary is the existing LLM narrative-analysis pass. Novelty tries already appear in it
  (they ride the `operation` column and their code is at `tries/try_{id:03d}/candidate.py`).
  The eye test requires editing the analysis prompt so that for `operation == novelty` tries
  it specifically assesses "is the core architecture genuinely different from the parent, or a
  cosmetic refactor?" No code change — but if this prompt edit is skipped, the discard-vs-
  refactor signal (the whole L2 bar) is muddy. It is a real deliverable, just a prompt one.

### L3 — Parent usage
**Parent-anchored, mid-search.** The operator takes a parent (a frontier member) so the run's
accumulated learning is available rather than starting cold. Note the *core model/architecture*
is not inherited — it is discarded (see L4). What survives as context is: (1) the parent's
**metrics**, which serve as the target/bar the novel solution must aim to beat, and (2) any
**side insights** the model judges still applicable. So "don't start from scratch" means the
performance bar and transferable learnings carry forward, not the code core.

This is the deliberate difference from multi-init: multi-init reseeds the *starting* population
cold; `novelty` fires *mid-search* with the run's learning (metrics bar + side insights) intact.
See **Purpose**.

### L4 — Anti-anchor mechanism (the core design)
The parent's **full candidate code is provided** — but the anchor it would otherwise create
is neutralized by a forced chain-of-thought the operator prompt imposes, in this order:
1. Distill the **core insight** of the parent solution.
2. Distill the **side insights**.
3. **Discard the core implementation** (throw away the existing model/architecture).
4. Build a **novel core** (new architecture/approach) for the same objective, aiming to beat
   the parent's metrics (which carry forward as the target bar — see L3).
5. Carry **side insights forward if still applicable** to the new core.

Applicability of side insights is left to **model discretion** — deliberately not
pre-specified. Rationale: a strong model (Sonnet/Opus) is expected to judge which learnings
transfer; over-engineering an applicability filter is premature before eye-testing whether
the model handles it. To be validated empirically.

**L4a — Context scope (what `build_novelty_feedback` injects vs. omits).** The existing
`build_improve_feedback` (py_mutation_engine.py:142) injects a rich context bundle: full
parent source, parent scores + confusion counts, best-so-far across the frontier, **neighbor
context** (`neighbor_lines`, `_NEIGHBOR_CAP`=5), and a **graveyard** of dead-ends (`dead_ends`,
py_mutation_engine.py:160–184, programmatic summaries, capped at 10). Novelty reuses only part
of this:
- **KEEP:** full parent source (required for the distill step) + parent metrics (target bar, L3).
- **DROP:** neighbor context and the graveyard. Rationale — these are *exploit* machinery: the
  neighbors are frontier members of the *current* family and the graveyard says "these
  tree-variants failed, don't repeat them," which implicitly frames the task as "find a better
  tree." Feeding them to an operator whose entire job is to *leave* the family re-anchors it to
  exactly what it's trying to escape. Cross-architecture, they are useless-to-harmful.
- **BUILD STYLE:** `build_novelty_feedback` is a **standalone builder** that reuses the
  low-level source-reader + a novelty CoT template, and simply does **not** call the
  neighbor/graveyard collectors. Deliberately *not* implemented by threading an
  `include_exploit_context=False` flag through the shared `build_improve_feedback`, because that
  couples novelty to a function three operators depend on and risks the working improve path.
  The small parent-source-assembly duplication is accepted to avoid that coupling.
- **T-phase guardrail:** a naive executor will copy the whole improve builder including
  neighbor/graveyard. The omission is deliberate and must be stated as a hard do-not in C2.


- **DOCUMENTED RISK (load-bearing):** this design keeps the full code (which is *precisely*
  the mechanism behind template lock-in) and relies on the CoT "discard" step to override
  the model's observed default of refactoring working tree code. This is a bet against a
  *documented* behavior, not a hypothesis. If the model performs the distillation as theater
  and then refactors anyway, **the operator degenerates into `improve`.** The L2 eye test is
  precisely the check for whether the discard step is honored.

### L5 — CoT structure
The full chain (L4) runs in a **single LLM call / single prompt.** Splitting it into three
sequential calls (distill → discard → rebuild) is a **deferred ablation** (see Out-of-scope)
— decided only after eye-testing the single-call version. Prove the mechanism works before
optimizing how the reasoning is structured.

### L6 — Trigger
Probability of firing the operator **decays linearly early, then plateaus at a floor** for
the rest of the run.
- **Shape rationale:** novelty is most valuable early (before the search commits to a basin)
  and least valuable late (refining a good frontier). The decay encodes explore-early /
  exploit-late. The **plateau floor** guarantees rare-but-nonzero late fires, which
  preserves eye-test coverage across the whole run (a pure decay-to-zero would starve the
  hard late-stage case — a mature parent is the toughest test of the discard step).
- **Axis = fraction of the active horizon**, where the horizon is whichever stop condition
  the run uses: fraction of max-tries under the tries flag, fraction of budget-spent under
  the budget flag. The trigger reads the same flag the run loop uses to stop, so the
  explore/exploit shape holds regardless of stopping mode and regardless of run size (20 vs
  50 tries behave consistently).
- **DOCUMENTED BEHAVIOR (not a defect):** under budget mode, spend-fraction is known only
  after each mutate call resolves and is uneven per try, so the decay **steps unevenly** and
  a single expensive try can vault past the crossover into the plateau. This is accepted;
  do not interpolate it smooth.

### L7 — Parameterization discipline
The trigger curve (L6) is **hardcoded for now** (no dynamic tuning), defined by three numbers
that must be **named, commented, module-level constants — never inline literals** (alongside
the existing `REPAIR_PROB` / `COMBINE_PROB` at runner.py:137–138):
- start probability (high, early-run)
- floor probability (the plateau)
- crossover point (fraction of horizon where decay meets the floor)

**These three are inputs to the decay function, not three independent gates.** Unlike the
existing operators — each of which rolls against a single flat constant (e.g.
`COMBINE_PROB = 0.3`) — `novelty` computes its per-try fire-probability from the
decay+plateau curve that these three constants parameterize (see L6 and the placement note).
Consequence for T-phase: the `novelty` stage in `choose_operation` is **not** a copy-paste of
the combine stage — it calls a probability-computing helper, then rolls against the result.

Values themselves are to be set at implementation (not chosen in this doc). The discipline —
named constants with a one-line rationale each — is a hard guardrail so tuning later is a
one-line edit and intent stays legible.

## L6 placement (resolved against real code)

`choose_operation` (runner.py:178) is a short-circuiting **waterfall**; constants at
runner.py:137–138:
1. **repair** — gate: `recent_failures` in last 5 archive entries (runner.py:152); fires at
   `REPAIR_PROB = 0.3` if gate met.
2. **combine** — gate: frontier ≥ 3; fires at `COMBINE_PROB = 0.3` if gate met; only reached
   if repair did not fire.
3. **improve** — default fallback; effective ~70% (no failures / small frontier) down to
   ~49% (failures present but repair/combine rolls missed).

**Decision:** insert `novelty` as a new waterfall stage **after combine (gate + roll), before
the improve fallback.**
- **Why:** preserves repair's priority (correctness recovery first) and combine's full
  effective fire-rate (your strongest Pareto operator, untouched). `novelty` can only fire
  when combine's gate fails (frontier < 3) or combine's roll misses — so it draws its
  budget **exclusively from improve**, never from combine. Net effect: exploration
  (combine + novelty) is up-weighted entirely at improve's expense, which is the intended
  explore/exploit rebalance.
- **Probability:** the `novelty` stage does **not** read a flat constant; it computes its
  per-try fire-probability from the L6 decay+plateau curve (see L7).
- **DOCUMENTED CONSEQUENCE (conditional fire-rate):** because the waterfall short-circuits,
  `novelty`'s *effective* fire-rate = `P_novelty(progress)` × P(repair didn't fire) ×
  P(combine didn't fire). Early in a run the frontier is often < 3 so combine's gate fails,
  leaving `novelty` and `improve` to share that mass — which *helps* early novelty and is
  consistent with the explore-early intent. But it means the nominal L7 start-probability is
  **not** the observed early fire-rate. **First-run verification compares actual novelty
  fires against the *conditional* expectation, not the nominal constant.**

## Out of scope (recorded so they're not lost)

- **3-call CoT split** (L5) — deferred ablation; revisit after single-call eye test.
- **Stagnation-based triggering** — a more principled trigger (fire when frontier hasn't
  improved in N tries) is deferred; current design uses the decay+plateau (L6).
- **Hypervolume-per-dollar grading** — the quantitative success harness is the *next
  project*; this operator is judged qualitatively (L2) first.
- **Dynamic parameter tuning** (L7) — constants are static for now.
- **`choose_operation` refactor** — unifying the waterfall into a per-operator abstraction
  (each operator declares its own gate + probability function) is deferred. `novelty` is
  added now as a minimal waterfall insert, not a copy-paste. Revisit trigger: once the
  ablation set (novelty / crowding / multi-init / islands) stabilizes — specifically after
  islands, since islands may dictate the selection/population interface. Refactoring now
  would target a moving interface and break one-change-at-a-time attribution.

## Documented risks (summary)

1. **Discard-step dependence (L4)** — the operator's whole value rests on the model honoring
   "throw away the core." If it doesn't, it degenerates into `improve`. Eye test is the check.
2. **Budget-mode step behavior (L6)** — decay is non-uniform under budget mode by
   construction; accepted.
3. **Conditional fire-rate (L6 placement)** — because `novelty` sits below repair and
   combine in the waterfall, its effective fire-rate is conditional on those not firing, so
   nominal start-prob ≠ observed rate. Verify actual vs *conditional* expected fire count in
   the first run.

- **Additive.** Do not modify `build_improve_feedback`, `build_combine_feedback`,
  `select_parent`, `pick_frontier_extremes`, `recent_failures`, or the repair path. Novelty

  single insert into `choose_operation`.

- **`EvaluationResult` is an object** — `.status` / `.metrics` / `.failure_reason` by

- **Constants are named, commented, module-level** — never inline literals (L7).
- **`build_novelty_feedback` deliberately OMITS neighbor + graveyard context** (L4a). A naive
  copy of the improve builder would include them; that is wrong here and is a hard do-not.

## Component 2 — Trigger probability (NEW)

### `novelty_fire_probability(progress_fraction: float) -> float`
- **Purpose:** map run progress → per-try novelty fire probability via linear decay to a floor.
- **Inputs:** `progress_fraction` in [0.0, 1.0] — fraction of the active horizon elapsed.
- **Output:** probability in [0.0, 1.0]. For `progress_fraction <= NOVELTY_CROSSOVER_FRAC`,
  linearly interpolate `NOVELTY_START_PROB → NOVELTY_FLOOR_PROB`; for `> NOVELTY_CROSSOVER_FRAC`,
  return `NOVELTY_FLOOR_PROB` (the plateau).
- **Side effects:** none (pure function).
- **Edge/failure:** clamp `progress_fraction` to [0,1] defensively is acceptable here since it
  is a computed input, but per primer "v1 raises loudly," prefer to assume the caller passes a
  valid fraction and let a bad value surface. **[DECISION — defer to human if unclear]:** clamp
  vs. assume-valid. Default recommendation: assume-valid (consistent with primer), no clamp.

### `run_progress_fraction(try_id: int, max_tries: int | None, tracker: CostTracker, budget: float | None) -> float`
- **Purpose:** compute elapsed fraction of the active horizon, branching on stop mode.
- **Behavior:** a run is bounded by **exactly one** of tries or budget (the CLI takes `--tries`
  *or* `--budget`, never both). So:
  - tries mode (`max_tries is not None`): `(try_id - 1) / max_tries` — `try_id` starts at 1 and
    increments at end of loop, so `try_id - 1` is the count *elapsed before* this try.
  - budget mode (`budget is not None`): `tracker.dollars() / budget` — `tracker` is the
    `CostTracker` accumulating Bedrock token spend across the run.
- **Output:** fraction in [0.0, 1.0]. Under budget mode this steps unevenly per try (L6,
  accepted — do NOT smooth/interpolate).
- **Edge/failure:** assumes exactly one bound is active. If **neither** is set, that is a broken
  run — **raise** (primer: v1 raises loudly), do not silently default to a fraction.
- **Side effects:** none (read-only).
- **[SCOUT — match, don't invent]:** confirm the live names — `try_id`, the `--tries` value,
  the `CostTracker` instance (`tracker`) and its `.dollars()` method, and `budget`. All confirmed
  present in the run loop (`runner.py` while-loop) and `CostTracker`; just thread the ones already
  in scope at the call site rather than adding new state.

## Component 5 — Novelty feedback builder (NEW — the heart)

### `build_novelty_feedback(parent_entry) -> str`
- **Purpose:** construct the novelty operator's feedback string: the forced-CoT instructions +
  full parent source + parent metrics as target bar. This string is passed to `mutate(...,
  feedback=...)`.
- **Inputs:** `parent_entry` — an archive entry dict (`try_id`, `candidate` Path, `result`
  EvaluationResult, `parent_try_id`). **[SCOUT]** confirm the archive-entry shape against real
  code and the exact means the existing builders use to read a candidate's source.
- **What it assembles (L4 / L4a):**
  1. Full parent source — read **inline**, matching the existing house pattern:
     `Path(parent_entry["candidate"]).read_text(encoding="utf-8")`. **Confirmed: there is NO
     shared source-reader helper** — `build_improve_feedback` (py_mutation_engine.py:104),
     `build_repair_feedback`, and `mutate` each read inline via `pathlib`. So `build_novelty_feedback`
     adds a fourth inline read in the same style; do NOT create a helper and do NOT refactor the
     existing three (see note below).
  2. Parent metrics (from `parent_entry` result `.metrics`) rendered as the **target bar** the
     novel solution must aim to beat.
  3. The **novelty CoT template** (PROMPT text, a module-level template constant): distill core
     insight → distill side insights → **discard the core implementation** → build a novel core
     (new architecture/approach) for the same objective → carry side insights forward *if
     applicable* (applicability = model discretion, L4).
- **DELIBERATELY OMITS (L4a, hard do-not):** neighbor context (`neighbor_lines`/`_NEIGHBOR_CAP`)
  and the graveyard (`dead_ends`). Do not call those collectors. This is the key difference from
  `build_improve_feedback` and the reason this is a **standalone** builder, not a flag on the
  shared one.
- **Output:** the feedback string.
- **Side effects:** reads a candidate source file from disk (same as existing builders); no writes.
- **Edge/failure:** if the parent source file is missing, that cannot happen in a correct run
  (parent came from a successful archive entry) → let it raise (primer: v1 raises loudly).
- **[SCOUT]** confirm the module this belongs in — the existing builders live in
  `py_mutation_engine.py`; `build_novelty_feedback` should sit alongside them for consistency.

# Build plan (C-phase) — 2 chunks for Gemini 3 Flash

**Why 2 and not 6:** this surface is small, so maximal chunking would pay re-orientation
overhead repeatedly without ever hitting the context-accumulation zone that makes chunking
pay off. But a single blob is risky for Flash — a long context invites "context rot" and
dropping a guardrail (esp. the L4a omission and the match-don't-invent scouts). The
break-even is a light split at the one real phase boundary: **pure new logic** vs
**integration with existing control flow.** Two chunks keeps each context small and each
guardrail salient, and makes the failure case cheap and re-runnable.

## Guardrails (state once, apply to both chunks)
- Additive only. Do NOT edit `build_improve_feedback`, `build_combine_feedback`,
  `select_parent`, `pick_frontier_extremes`, `recent_failures`, or the repair path.
- All randomness threads the shared `_rng`. Never a new RNG, never bare `random.*`.
- No renaming existing keys / labels / fields / constants.
- `EvaluationResult` is an object (`.status` / `.metrics` / `.failure_reason`).
- Constants are named, commented, module-level — never inline literals.
- **[MATCH, DON'T INVENT]** at every touch of existing code: read the surrounding code and
  match its names/conventions; on any conflict with this recipe, STOP and report.
- v1 raises loudly — no defensive swallowing of things that can't happen in a correct run.

## Chunk 0 — Orientation (fresh session context)
Read the novelty operator **design doc** (L1–L7, L4a) and the SensorFlow Primer first.
Background in one paragraph: SensorFlow is an LLM-guided evolutionary search over TinyML
candidates; operators (`improve`/`combine`/`repair`) mutate a parent candidate via a paid
Bedrock `mutate()` call. This work adds a fourth operator, `novelty`, that makes the LLM
produce a *completely novel solution* (new architecture) instead of refining — to break the
observed DecisionTree template lock-in. Judged by eye test, not metrics. Files in play:
`runner.py` (loop, `choose_operation`, constants, `select_parent`) and
`py_mutation_engine.py` (feedback builders, `mutate`).
