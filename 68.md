# Triage: missing `reasoningContent` from Claude on Bedrock Converse API

> **For a coding agent.** Goal: diagnose why a raw-boto3 `converse` call to
> `us.anthropic.claude-opus-4-8` returns text but no reasoning/thinking block. Work the steps
> **in order** and STOP at the first one that identifies the cause — do not apply all fixes
> blindly. Report which step resolved it. Do not refactor unrelated code.

## Context / known facts
- Model: `us.anthropic.claude-opus-4-8` (Claude 4.x — DOES support extended thinking on Bedrock).
- Transport: raw boto3 `bedrock-runtime` `converse` (NOT a framework — so framework-drop bugs
  are ruled out).
- Symptom: response returns normally (no exception) but contains no reasoning content.
- The real Converse schema nests reasoning as:
  `content_block["reasoningContent"]["reasoningText"]["text"]` and `[...]["reasoningText"]["signature"]`
  — one level deeper than the commonly-assumed flat `["reasoningContent"]["text"]`.

## Step 0 — Capture the raw response shape (do this first, always)
Add a diagnostic print immediately after the `converse` call:
```python
content = resp["output"]["message"]["content"]
print("BLOCK_KEYS:", [list(b.keys()) for b in content])
```
Branch on the output:
- Output includes `reasoningContent` (e.g. `[['reasoningContent'], ['text']]`)
  → the API IS returning reasoning. Cause is **parsing**. Go to **Step A**. STOP after fixing.
- Output is text-only (e.g. `[['text']]`)
  → reasoning was never generated. Cause is **request config**. Go to **Step B**.
- The `converse` call itself raises (ValidationException / AccessDenied / ResourceNotFound)
  → not a reasoning problem at all. Go to **Step C**.

## Step A — Parsing bug (reasoning block present but not surfaced)
The block exists; the code reads the wrong nesting level.
- Find where the code reads reasoning. If it does `block["reasoningContent"]["text"]` or
  `block["reasoningContent"].get("text")`, that is the bug — it reads one level too shallow and
  yields empty/KeyError on real responses.
- Correct access:
  ```python
  for block in content:
      if "reasoningContent" in block:
          rc = block["reasoningContent"]
          thinking_text = rc.get("reasoningText", {}).get("text", "")
          signature     = rc.get("reasoningText", {}).get("signature")
          redacted      = rc.get("redactedContent")   # encrypted thinking, if any
      elif "text" in block:
          answer_text = block["text"]
  ```
- STOP. Report "parsing / wrong nesting level" as the cause.

## Step B — Request config (no reasoning generated)
Reasoning is off. Two independent causes — check BOTH before concluding:

**B1 — thinking flag not passed / wrong field name.**
- Confirm the request includes:
  ```python
  additionalModelRequestFields={"thinking": {"type": "enabled", "budget_tokens": 2000}}
  ```
- Common mistake: using the legacy key `reasoning_config` instead of `thinking`. The newer model
  silently ignores the unknown field and returns text only. Use `thinking`.
- `budget_tokens` must be a positive integer and less than `maxTokens`.

**B2 — temperature not 1.0.**
- Extended thinking requires temperature = 1.0. If `inferenceConfig` sets any other temperature
  (or a leftover default like 0.7), the enable can fail silently → text only.
- Fix: `inferenceConfig={"temperature": 1.0, "maxTokens": <N greater than budget_tokens>}`.
- Also remove/avoid `topP`/`top_k` constraints that conflict with thinking if present.

After correcting B1 and B2, re-run Step 0. If `reasoningContent` now appears, STOP and report
which of B1/B2 was the cause (or both).

## Step C — Call errors before generation (not a reasoning issue)
- **ValidationException** → inspect the message. Often `budget_tokens >= maxTokens`, or an
  inference param incompatible with thinking (temperature ≠ 1.0 can surface here too).
- **AccessDeniedException / ResourceNotFound** → model-ID / access problem, unrelated to
  reasoning. Confirm the exact identifier your account has enabled: the cross-region inference
  profile (`us.anthropic.claude-opus-4-8`) vs the plain model ID (`anthropic.claude-opus-4-8`)
  behave differently for access. Verify the enabled ID/ARN in the Bedrock console and match it.
- STOP. Report the error class and the offending field.

## Minimal reference call (known-good shape)
```python
resp = client.converse(
    modelId="us.anthropic.claude-opus-4-8",   # confirm vs your enabled profile/ARN
    messages=[{"role": "user", "content": [{"text": "PROMPT"}]}],
    additionalModelRequestFields={"thinking": {"type": "enabled", "budget_tokens": 2000}},
    inferenceConfig={"temperature": 1.0, "maxTokens": 4000},
)
content = resp["output"]["message"]["content"]
print("BLOCK_KEYS:", [list(b.keys()) for b in content])
```

## Multi-turn / tool-use note (only if applicable)
If reasoning is enabled AND the flow uses tool use: on the follow-up request, the assistant turn
that carried a `toolUse` block MUST also carry the preceding signed `reasoningContent` block
(with its original `reasoningText.text` and `signature`, unmodified) or Bedrock rejects the
replay with a ValidationException. If you rebuild assistant turns from text+tool_calls only,
re-emit the reasoningContent block ahead of the toolUse block. (Not applicable to single-shot
calls with no tools.)

State: (1) which step identified the cause, (2) the specific field/line at fault, (3) the one
change that fixed it, (4) confirmation that Step 0's print now shows `reasoningContent`. Do not
change anything outside the identified cause.
